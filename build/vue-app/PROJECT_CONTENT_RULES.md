## 二、详细章节规范

每个项目详情页面必须按以下顺序包含以下章节：

### 1. 项目标题（H1）
```markdown
# 项目名称 - 简短标语
```

**要求：**
- 使用一级标题
- 格式：`项目名称 - 简短标语`
- 标语不超过10个字，突出项目核心价值

---

### 2. 项目介绍（H2）
```markdown
## 项目介绍

### 项目概述
[3-5句话介绍项目是什么，解决什么问题]

### 核心功能
- [用列表形式列出3-8个核心功能点]
- [每个功能点简明扼要]

### 使用场景
[描述项目适用的场景和目标用户]
```

**要求：**
- 使用二级标题
- 包含项目概述、核心功能、使用场景三个子部分
- 核心功能使用无序列表，每项不超过20字

---

### 3. 技术栈（H2）
```markdown
## 技术栈

### 核心技术
| 技术组件 | 版本/说明 | 用途 |
|----------|-----------|------|
| 编程语言 | 版本号 | 主要开发语言 |
| 框架 | 版本号 | GUI/后端框架 |
| 构建工具 | 版本号 | 项目构建 |

### 依赖库
- **主要依赖**：列出主要第三方库
- **可选依赖**：列出可选功能依赖

### 开发环境
- IDE/编辑器推荐
- 必要的开发工具
```

**要求：**
- 使用表格展示核心技术
- 包含技术组件、版本/说明、用途三列
- 至少列出3项核心技术

---

### 4. 系统要求（H2）
```markdown
## 系统要求

### 最低配置
| 项目 | 要求 |
|------|------|
| 操作系统 | 具体版本 |
| 处理器 | 最低CPU要求 |
| 内存 | 最低RAM要求 |
| 存储空间 | 最低硬盘空间 |
| 显示器 | 分辨率要求（如有） |

### 推荐配置
| 项目 | 要求 |
|------|------|
| 操作系统 | 推荐版本 |
| 处理器 | 推荐CPU |
| 内存 | 推荐RAM |
| 存储空间 | 推荐硬盘空间 |
| 显示器 | 推荐分辨率（如有） |
```

**要求：**
- 必须包含最低配置和推荐配置两个表格
- 每个表格至少包含操作系统、处理器、内存、存储空间四项
- 使用具体版本号，避免模糊描述

---

### 5. 硬件要求（H2）

- 不提供历史版本和其他下载方式

## 三、编写规范

### 格式要求
1. **标题层次**: 严格按照规定的H1、H2、H3结构
2. **列表使用**: 功能点使用无序列表(-)，步骤使用有序列表(1.)
3. **表格使用**: 系统要求和技术栈必须使用表格
4. **链接格式**: 使用标准Markdown链接格式 `[文本](链接)`
5. **代码块**: 代码示例使用三个反引号并标明语言

### 内容要求
1. **准确性**: 所有信息必须准确无误，特别是版本号、日期、链接
2. **完整性**: 每个章节必须包含规定的最低信息量
3. **一致性**: 同类项目使用相同的描述风格和格式
4. **时效性**: 更新记录和开发计划必须保持最新
5. **可验证性**: 所有外部链接必须可访问

### 语言要求
1. **简洁明了**: 使用简洁清晰的语言，避免冗长
2. **专业准确**: 技术术语使用准确，避免模糊描述
3. **中文为主**: 主要内容使用中文，代码和专有名词除外
4. **避免广告**: 描述客观真实，避免过度营销语言

## 四、文件命名规范

### 项目详情页文件
- **格式**: `项目名称.md`
- **示例**: `english-listen.md`
- **位置**: `project/` 目录下

### 辅助文件
- **更新记录**: `update.md`（可选，项目内使用相对链接）
- **开发计划**: `todo.md`（可选，项目内使用相对链接）
- **图片资源**: `assets/项目名称/` 目录下

## 六、维护要求

1. **定期更新**: 每次版本发布后必须更新更新记录
2. **链接检查**: 每月检查一次所有外部链接的有效性
3. **内容审查**: 每季度审查一次内容的准确性和时效性
4. **规范更新**: 根据实际需要适时更新本规范
